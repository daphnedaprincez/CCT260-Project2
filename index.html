Process

Summary report: phases from planning through accessibility checks, with visuals for inspiration and desktop layout.

1. Planning

Goals: one index.html file with five logical areas (home, portfolio, resume, contact, process), a sticky header with navigation that jumps to each section, main content plus a secondary sidebar and footer; external CSS only; bio, photo, LinkedIn, embedded videos, tabular resume, contact form with rich media, and this summary including wireframe and WCAG notes.

Constraints: static HTML, no inline styling for layout or theme, semantic regions for assistive technologies, and content aligned with the assignment rubric.

2. Rationale

A pure black base with neon pink, purple, and blue keeps hierarchy obvious and matches the reference portfolio mood. Pink project cards and a glowing circular portrait echo arcade-inspired UI; the sidebar repeats anchor links for a second path besides the top navigation.

3. Inspirations

The board image below informed spacing, emphasis on a single accent, and a calm editorial tone for headings versus UI chrome.

4. Information architecture (IA)

  • Home (#home): identity (bio + photo), LinkedIn, highlight cards to other sections, primary video embed.
  • Portfolio (#portfolio): three representative work blocks.
  • Resume (#resume): tables for education, experience, and skills with captions and scoped headers.
  • Contact (#contact): labeled form, second video, inspiration image.
  • Process (#process): narrative sections, inspiration figure, desktop wireframe, WCAG summary.

5. Graphic design

Black canvas with neon pink, purple, and electric blue echoes a retro arcade mood. Press Start 2P carries section titles; Varela Round keeps body copy friendly and readable. Gradient text, glowing profile frame, and pill buttons mirror the reference portfolio aesthetic.

6. Desktop wireframe — page structure

Low-fidelity wireframe (desktop width) showing header, horizontal nav with in-page links, two-column main + sidebar, and footer. Regions match this single-page layout.

7. WCAG 2.0 accessibility — measures taken

The following practices align with common WCAG 2.0 Level A and AA themes (perceivable, operable, understandable, robust):

  • 1.1.1 Non-text content: meaningful alt on images; decorative regions marked with aria-hidden where appropriate.
  • 1.3.1 Info and relationships: semantic header, nav, main, aside, footer; table caption, th with scope.
  • 1.4.3 Contrast (minimum): light text on dark backgrounds and accent links tuned for readable contrast in primary states.
  • 2.1.1 Keyboard: all interactive elements are native links, buttons, or focusable embeds; skip link targets main content.
  • 2.4.1 Bypass blocks: “Skip to main content” link at the top of the page.
  • 2.4.7 Focus visible: visible :focus-visible outlines on links, buttons, and form controls.
  • 3.3.2 Labels or instructions: every form control has an associated label; iframe titles describe embedded videos.
  • 4.1.2 Name, role, value: aria-label / aria-labelledby on nav and regions where headings clarify purpose.

Ongoing: run automated checks (e.g. axe, Lighthouse) and manual keyboard-only passes before submission; adjust contrast or focus styles if audit tools flag edge cases.
